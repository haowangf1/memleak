# 析构函数为什么会因跨模块绑定而错误释放内存

  ## 1. 结论

  析构函数不是抽象概念，而是一段已经编译好的具体代码。

  也就是说：

  - 析构函数这个符号对应一段机器码
  - 这段代码里已经写明了：
    - 读取对象哪些成员
    - 哪些成员要释放
    - 哪些成员只析构但不释放
    - 是否调用其他子析构或辅助函数

  因此，一旦运行时把析构调用绑定到某个模块里的那份实现，就会**直接按那份实现的代码执行**。

  如果对象实际上是按另一个模块的实现语义创建的，就可能出现：

  - 对成员解释错误
  - 错误释放不该释放的地址
  - `Invalid free`
  - 崩溃

  ---

  ## 2. 本质原理

  问题不在于：

  - 两个模块的内存是否重叠

  而在于：

  - 同一块对象内存
  - 被哪一份析构代码来解释和处理

  换句话说：

  - 对象内存没变
  - 但“处理这块内存的代码”变了
  - 一旦代码和对象的实际构造语义不一致，就会出错

  ---

  ## 3. 为什么“绑定到哪份析构实现”很重要

  析构函数本质上是一段逻辑，例如可能会做这些事：

  - 读取 `this->member1`
  - 如果非空则 `free(member1)`
  - 读取 `this->member2`
  - 调用 `member2` 的析构
  - 跳过某些仅借用的静态地址
  - 清理本模块内部维护的状态

  所以：

  - 析构调用绑定到哪个符号地址
  - 就决定了由哪份逻辑来处理对象

  如果这份逻辑不是对象原始构造时所对应的那套实现，释放策略就可能错。

  ---

  ## 4. 一个简化例子

  ## 假设对象长这样

  ```cpp
  class Formatter {
  public:
      char* heap_buf;          // 堆内存
      const char* static_pat;  // 指向模块内只读常量
      ~Formatter();
  };

  对象在内存里可以粗略理解为：

  Formatter object
    +-------------------------+
    | heap_buf   = 0x500000   |
    | static_pat = 0x7f....   |
    +-------------------------+

  其中：

  - heap_buf 指向堆内存，可以合法释放
  - static_pat 指向模块内部只读段，不可释放

  ———

  ## 5. NextMesh 里的析构实现

  Formatter::~Formatter() {
      if (heap_buf) {
          free(heap_buf);
      }

      // static_pat 指向模块内只读常量，不释放
  }

  这份析构逻辑可以理解为：

  - 释放 heap_buf
  - 不处理 static_pat

  编译后对应的机器码逻辑大致就是：

  load this->heap_buf
  if not null -> call free
  return

  也就是说，这份析构“说明书”明确写的是：

  - heap_buf 释放
  - static_pat 不释放

  ———

  ## 6. MCNP 里的另一份析构实现

  假设另一模块里，因为实现差异、内部逻辑差异或编译条件差异，析构写成了：

  Formatter::~Formatter() {
      if (heap_buf) {
          free(heap_buf);
      }

      if (static_pat) {
          free((void*)static_pat);   // 错误地把它当成动态资源
      }
  }

  这份析构对应的机器码逻辑大致是：

  load this->heap_buf
  if not null -> call free

  load this->static_pat
  if not null -> call free

  return

  也就是说，这份析构“说明书”写的是：

  - heap_buf 释放
  - static_pat 也释放

  ———

  ## 7. 正常情况不会出问题

  如果对象在 NextMesh 中创建：

  Formatter* p = new Formatter;
  p->heap_buf = (char*)malloc(128);
  p->static_pat = "PATTERN_X";   // 指向 NextMesh.so 的只读段

  然后析构时也执行 NextMesh 那份析构：

  delete p;

  那么实际效果是：

  free(p->heap_buf);    // 合法
  // p->static_pat 不处理

  所以不会出问题。

  ———

  ## 8. 出问题的情况：析构调用绑定到了另一模块

  问题发生在运行时符号绑定错位时。

  假设对象仍然由 NextMesh 创建：

  Formatter* p = new Formatter;
  p->heap_buf = (char*)malloc(128);
  p->static_pat = "PATTERN_X";   // 指向 NextMesh.so 的 r-- 段

  但执行：

  delete p;

  时，真正调用到的却不是 NextMesh 的析构，而是 MCNP 模块中的那份析构实现。

  这时就会执行：

  free(p->heap_buf);     // 合法
  free(p->static_pat);   // 非法

  因为：

  - p->static_pat 实际不是堆地址
  - 它指向的是 NextMesh.so 的只读段
  - 所以执行 free 会触发：

  Invalid free

  ———

  ## 9. 这说明了什么

  这说明问题不是：

  - “数据本身被重定位错了”

  而是：

  - “处理这份数据的析构代码入口被绑定错了”

  也就是说：

  - 对象内容没有变
  - 但执行的是错误的析构逻辑
  - 因此同一块对象内存被用错误规则解释了

  ———

  ## 10. 为什么可以说“析构说明书已经写死”

  因为编译完成后，析构函数就是一段固定机器码。

  例如逻辑上的：

  if (heap_buf) free(heap_buf);

  最终会变成类似：

  读取 this + offset(heap_buf)
  判空
  调用 free

  所以可以把析构理解成一份已经写死的“说明书”：

  - 读哪个成员
  - 调什么释放函数
  - 哪个成员跳过
  - 哪个成员递归调用子析构

  一旦程序运行时把析构调用绑定到某个模块的这份代码，就会直接按这份“说明书”执行。

  它不会知道：

  - 这个对象其实是别的模块按另一套规则构造出来的

  ———

  ## 11. 对应到实际崩溃问题

  在实际案例中：

  - NextMesh 创建/设置了 spdlog::pattern_formatter/full_formatter
  - 对象内部可能引用了 NextMesh 模块中的只读常量或静态表
  - 但析构时调用链落到了 MCNP 中的 spdlog 析构实现
  - MCNP 那份析构代码错误地处理了这些内部指针
  - 最终对 NextMesh.so 的 r-- 段地址执行 free
  - 于是出现 Invalid free 和崩溃

  ———

  ## 12. 一句话总结

  析构函数本质上是一段已经写死“如何处理对象成员”的代码。

  因此：

  - 析构调用绑定到哪份实现
  - 就决定了谁来解释这块对象内存
  - 如果对象和析构代码不属于同一套实现语义
  - 就可能错误释放数据并崩溃
